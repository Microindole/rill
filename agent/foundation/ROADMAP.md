# 路线图

## 总体规划

本项目后续按以下大流程推进，所有 agent 默认以此为准。

### 0. 修复启动入口

目标：

- 启动方式跨平台
- 运行方式统一
- 易部署
- 减少对 IDEA 私有运行配置的依赖

重点内容：

- 统一 launcher
- 收敛多个 `main` 入口
- 明确正式入口与工具入口
- 统一命令风格与参数风格
- 保证终端、VS Code、打包运行一致

### 1. 重构底层代码

目标：

- 重构整个项目底层代码
- 使用更合理的设计模式
- 提升模块边界、可维护性与可扩展性

重点内容：

- 梳理核心抽象
- 降低耦合
- 清理实验性实现
- 统一运行期对象组织方式
- 为后续功能扩展打基础

### 2. 收敛客户端入口

目标：

将客户端入口明确分为四类：

- CLI
- GUI
- Web UI
- Navicat 兼容入口

说明：

- CLI：面向终端与开发调试
- GUI：面向本地桌面交互
- Web UI：面向部署后的展示与演示
- Navicat 兼容：面向数据库客户端工具接入

### 3. 加入 Web 页面设计并与 Spring Boot 结合

目标：

- 建立可部署展示的 Web UI
- 将网页能力与 Spring Boot 运行模式结合

说明：

这一阶段不只是做前端页面，而是明确 Web UI 在整个系统中的角色和访问路径。

### 4. 引入网络编程、Redis 等，并使用 submodule

目标：

- 引入网络编程能力
- 引入 Redis 等外围基础设施
- 按子模块方式组织部分新能力
- 在 Git 层面也体现为更清晰的模块边界

说明：

这里的方向更接近“类似微服务的模块拆分”，但必须谨慎，避免为了拆分而拆分。

### 5. 自动化测试、管理与 CI/CD 工作流

目标：

- 补强自动化测试
- 补强工程管理能力
- 建立 CI/CD 工作流
- 部署流程放在最后完成，但质量体系应逐步补齐

说明：

这一阶段是最后成型阶段，但相关能力会在前面逐步完善，而不是全部拖到最后。

## 执行原则

- 先稳定启动和运行模型，再做大功能改造
- 先稳定架构边界，再做外延扩展
- 所有新增功能必须服务于“系统型 Java 项目”的目标
- 不为了看起来复杂而盲目引入分布式或微服务概念

## 当前优先级

当前最近优先事项：

1. 继续收口编译器可扩展性与 PostgreSQL 方言
2. 继续压实多模块、发布边界和安装模型
3. 为后续分布式能力预留稳定接口和模块位置
4. 再推进更完整的 Web UI 与网络化能力
5. 继续补齐 CI/CD 与完整部署体系
